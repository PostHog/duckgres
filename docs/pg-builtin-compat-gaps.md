# PostgreSQL builtin compatibility gaps (DuckDB) — audit modeled on PR #705 (array_lower)

Generated by a multi-agent hunt: 100 candidates empirically verified against bare DuckDB v1.5.2. 81 real gaps pre-dedup; 66 synthesized issues; 19 discarded (native_ok/already_covered/inconclusive).

## Summary

Audited 60+ candidate PostgreSQL functions/operators against bare DuckDB v1.5.2 and the duckgres source. After deduping by name (keeping highest-severity evidence) the confirmed real gaps collapse to 49 distinct issues, plus 19 discarded (native_ok / already_covered / inconclusive). Ranking is by real-world blast radius: client/driver STARTUP breakers and silent-wrong-data divergences first, then application-SQL hard-errors, then niche helpers. The single most dangerous finding is set_config — JDBC/SQLAlchemy/psycopg/poolers emit SELECT set_config(...) at connect time, so it breaks sessions before any query runs; it is trivially macro-able. uuid_generate_v4 and the jsonb containment operator @> are the next-broadest (ubiquitous ORM/schema patterns). Two SILENT-data traps deserve special note because duckgres currently makes them WORSE: format(), encode(), decode(), substr, substring, and width_bucket are listed as identity ('same') passthroughs in transpiler/transform/functions.go functionNameMapping, but the underlying DuckDB builtin is either absent (width_bucket -> Catalog Error) or has incompatible semantics (format uses {} not %I/%L; 2-arg decode returns input unchanged; substr clamps negative offsets). Those identity entries are no-op mis-mappings that should be replaced by real macros/transforms. The pg_get_function_* family (arguments/result/identity_arguments) collectively breaks psql \df and is independently confirmed broken in duckgres' own QA rawdata. Set-returning functions (json/jsonb_array_elements, json/jsonb_each, jsonb_path_query, aclexplode, pg_identify_object, pg_options_to_table, pg_get_keywords) and range/operator gaps (@>, #>>, OVERLAPS, daterange/numrange, the {1,2,3} array-text cast) are 'needs design' because a scalar CREATE MACRO cannot return a table or define an operator/type — they require a table macro, an A_Expr rewrite in operators.go, or a type shim. Honest bottom line: DuckDB already covers the large majority of PG's scalar surface; the genuine, macro-fixable wins are concentrated in ~15 scalar functions, and the highest-leverage single change is set_config.

## HIGH severity (7)

### set_config  — missing
- **Signature:** set_config(setting_name text, new_value text, is_local boolean) -> text
- **Evidence:** $ duckdb -c "SELECT set_config('search_path','main',false)" -> Catalog Error: Scalar Function with name set_config does not exist! Did you mean "set_bit"?. The 2-arg form errors identically. SET ... is a statement and cannot be used inside SELECT the way drivers emit it. The read-side twin current_setting() exists. duckgres covers current_setting (catalog.go:978) but has no set_config macro (grep empty). DuckDB v1.5.2.
- **Impact:** JDBC, SQLAlchemy (execution_options/search_path init), psycopg2 connection setup, pgbouncer-style poolers, and Rails/Django RLS multitenancy all emit SELECT set_config('search_path'|'app.current_tenant'|'statement_timeout', $1, false) so it can run in the data/extended-query path instead of via SET. Every one of these hits a Catalog Error at SESSION/CONNECTION STARTUP, breaking the client before any query runs. Broadest startup blast radius of any finding.
- **Fix (macro):** CREATE OR REPLACE MACRO set_config(setting_name, new_value, is_local) AS (SELECT new_value);  -- NOTE: a pure macro cannot actually mutate the session GUC the way PG's set_config does; it returns the requested value so the SELECT succeeds and tools proceed. For settings duckgres genuinely tracks (search_path), the real fix is a small special-case transform in setshow.go that rewrites SELECT set_config('search_path', v, local) into a SET search_path = v plus returning v, so the side effect actually happens. Marked needs-design-for-side-effect but the value-returning macro unblocks startup immediately.
- **Classify token:** SET_CONFIG(
- **Qualify map:** "set_config": true,  // session setting writer (value-returning compat macro)
- **Regression test:** Assert SELECT set_config('search_path','main',false) returns the text 'main' (one row, one column) without error; and, if the setshow transform is added, that current_setting('search_path') reflects 'main' afterward.

### uuid_generate_v4  — missing

### @>  — missing
- **Signature:** jsonb @> jsonb -> boolean (jsonb containment operator)
- **Evidence:** $ duckdb -c "SELECT '{"a":1,"b":2}'::json @> '{"a":1}'::json" -> Binder Error: No function matches '@>(JSON, JSON)'. Candidate: @>(T[],T[]) -> BOOLEAN. Native json_contains('{"a":1,"b":2}','{"a":1}') -> true; json_contains(...,'{"a":2}') -> false. duckgres operators.go handles only -> and ->> for JSON (lines 287-290); the @>/json_contains mapping is only a doc comment (lines 553-554,569) with no A_Expr case, so @> passes through and fails at the DuckDB binder.
- **Impact:** The primary GIN-indexed jsonb containment filter in PostgreSQL, used pervasively: Django JSONField __contains, SQLAlchemy .contains(), ActiveRecord, and hand-written WHERE col @> '{...}'. Every such query hits a hard Binder Error.
- **Fix (macro):** needs design — @> is an OPERATOR, not a callable function, so CREATE MACRO cannot define it. Fix is an A_Expr rewrite in transpiler/transform/operators.go: when the operator is '@>' and an operand is JSON-typed, rewrite (a @> b) to json_contains(a, b). DuckDB's json_contains already matches PG containment semantics for the common object/array cases.
- **Classify token:** @>
- **Qualify map:** n/a — operator handled in operators.go A_Expr switch, not a qualified function name (no pgcatalog.go map entry).
- **Regression test:** Assert SELECT '{"a":1,"b":2}'::json @> '{"a":1}'::json is true and ...@> '{"a":2}'::json is false after transpilation to json_contains.

### format  — divergent
- **Signature:** format(formatstr text [, formatarg ...]) -> text
- **Evidence:** $ duckdb -c "SELECT format('%I = %L','foo bar','baz')" -> '%I = %L' (template returned literally; args silently dropped). format('{} = {}','foo bar','baz') -> 'foo bar = baz' (DuckDB uses Python {} placeholders). format('%s and %%','hello') -> '%s and %%' (no %-substitution, %% not unescaped). DuckDB v1.5.2. NOTE: functions.go to_char already routes number-format through DuckDB format(), and there is no PG-%-style format() macro.
- **Impact:** PL/pgSQL dynamic SQL and migration/ORM scripts that build identifiers/literals via format('%I',...)/format('%L',...) receive the UNSUBSTITUTED template string with NO error raised. Silently-wrong SQL: at best a syntax error downstream, at worst identifiers/literals left unquoted, defeating %I/%L injection-safe quoting. Silent corruption is the dangerous failure mode here.
- **Fix (macro):** needs design — DuckDB's format() already exists with incompatible {}-placeholder semantics, so CREATE OR REPLACE MACRO format(...) would collide/recurse and cannot do variadic printf parsing in pure SQL. Fix is a special-function transform (functions.go specialFunctions) that, when the first arg is a literal containing %I/%L/%s, rewrites to a quote_ident/quote_literal/CAST-AS-VARCHAR concatenation chain (printf_to_concat), leaving {}-style calls untouched. Variadic + non-literal templates need an AST rewrite, not a macro.
- **Classify token:** FORMAT(
- **Qualify map:** n/a for macro — but if a compat format wrapper is added it must be qualified; otherwise handled as a specialFunctions transform (no pgcatalog map entry).
- **Regression test:** Assert format('%I = %L','foo bar','baz') yields the PG string "foo bar" = 'baz' (double-quoted identifier, single-quoted+escaped literal), and format('%s and %%','hello') yields 'hello and %'.

### decode  — divergent
- **Signature:** decode(string text, format text) -> bytea
- **Evidence:** $ duckdb -c "SELECT length(decode('YWJj','base64'))" -> 4 (PG=3); decode('YWJj','base64') -> 'YWJj' type VARCHAR (input returned UNCHANGED, format arg ignored); decode('616263','hex') -> '616263' len 6 (PG 'abc' len 3). Native from_base64('YWJj') -> BLOB len 3; unhex(...) -> BLOB. duckgres functions.go maps decode->decode as identity ('same', line 123), so the broken builtin is passed straight through.
- **Impact:** Any client/ETL round-tripping binary via encode()/decode() (base64 transport, hex blobs, COPY/migration scripts, ORMs with bytea columns) gets SILENTLY-WRONG data: the encoded string returned unchanged, wrong type (VARCHAR not bytea), wrong length, no error. The existing identity mapping makes duckgres complicit in the silent corruption.
- **Fix (macro):** CREATE OR REPLACE MACRO decode(s, fmt) AS CASE lower(fmt) WHEN 'base64' THEN from_base64(s) WHEN 'hex' THEN unhex(s) WHEN 'escape' THEN encode(s) ELSE error('unsupported decode format: ' || fmt) END  -- returns BLOB matching PG bytea. Also REMOVE the identity 'decode':'decode' entry from functions.go functionNameMapping so the macro is reached.
- **Classify token:** DECODE(
- **Qualify map:** "decode": true,  // 2-arg PG bytea decoder (base64/hex/escape) macro
- **Regression test:** Assert octet_length(decode('YWJj','base64'))=3 and decode result equals 'abc'::blob; octet_length(decode('616263','hex'))=3. Also assert functions.go no longer identity-maps decode.

### pg_get_function_arguments  — missing
- **Signature:** pg_get_function_arguments(func oid) -> text
- **Evidence:** $ duckdb -c "SELECT pg_get_function_arguments(0)" -> Catalog Error: Scalar Function with name pg_get_function_arguments does not exist! Did you mean "pg_get_constraintdef". duckdb_functions() has only pg_get_constraintdef/pg_get_expr/pg_get_viewdef in the pg_get family; 0 rows match %function_arg%/%get_function%. duckgres' own docs/iceberg-pg-syntax-qa-rawdata.json independently confirms \df fails because this family is unimplemented.
- **Impact:** psql \df and \df+ (describeFunctions in src/bin/psql/describe.c) SELECT pg_catalog.pg_get_function_arguments(p.oid) FROM pg_proc; the missing symbol aborts the ENTIRE query with a Catalog Error, so \df returns nothing. pgAdmin4 function node, DBeaver procedures view, JDBC getFunctionColumns, and pg_dump CREATE FUNCTION reconstruction all break.
- **Fix (macro):** CREATE OR REPLACE MACRO pg_get_function_arguments(oid) AS ''  -- empty string is a valid PG return (matches a no-arg function) and prevents the Catalog Error; duckgres has no real pg_proc rows to derive a true signature from, so empty-text is the safe, lossless stub.
- **Classify token:** PG_GET_FUNCTION_ARGUMENTS(
- **Qualify map:** "pg_get_function_arguments": true,  // function-arg signature stub for \df
- **Regression test:** Assert SELECT pg_get_function_arguments(0) returns '' (text, not error), so psql \df's join over pg_proc no longer aborts.

## MEDIUM severity (22)

### substr  — divergent
- **Signature:** substr(string text, start int [, count int]) -> text
- **Evidence:** $ duckdb -c "SELECT substr('alphabet',-2,5)" -> 'et'; PG14 -> 'al'. substr('alphabet',-3): DuckDB 'bet', PG 'alphabet'. Zero-start substr('alphabet',0,3) matches both ('al'). DuckDB clamps a negative start to position 1; PG treats it as a true origin and intersects window [start,start+count) with valid positions. functions.go maps substr->substr identity (line 36), passing the divergent builtin through.
- **Impact:** Queries passing a negative start offset to substr() get SILENTLY DIFFERENT substrings (no error). Affects application SQL deliberately relying on PG's arithmetic-window semantics. Narrow blast radius (no introspection tool uses negative offsets) but a data-correctness divergence the current identity mapping does nothing to fix.
- **Fix (macro):** CREATE OR REPLACE MACRO pg_substr(s, start, cnt) AS substr(s, greatest(start,1), greatest(0, cnt + start - greatest(start,1)))  -- emulates PG's window intersection for negative start. Cannot CREATE MACRO over the builtin name substr without recursion; implement as a functions.go specialFunctions rewrite of substr(s,start,count) -> the clamped-window expression, and likewise drop the identity substr mapping.
- **Classify token:** SUBSTR(
- **Qualify map:** n/a for macro — handled as a specialFunctions AST rewrite (no pgcatalog map entry); remove identity entry from functionNameMapping.
- **Regression test:** Assert substr('alphabet',-2,5)='al', substr('alphabet',-3)='alphabet', substr('alphabet',0,3)='al' (regression matrix vs PG14 ground truth).

### substring  — divergent
- **Signature:** substring(string text FROM pattern text) -> text
- **Evidence:** $ duckdb -c "SELECT substring('Thomas' from '...$')" -> Conversion Error: Could not convert string '...$' to INT64 (DuckDB treats the regex pattern as a start position). substring(x FROM int FOR int) works ('mas'). Native regexp_extract('Thomas','...$') -> 'mas'. functions.go maps substring->substring identity (line 37).
- **Impact:** Any PG/ORM/data-cleaning query using the SQL-standard substring(col FROM 'regex') extraction idiom (and the SIMILAR substring(x FROM pat FOR escape) form) hits a hard Conversion Error instead of the extracted substring. Hard failure (not silent), but the identity mapping leaves it broken.
- **Fix (macro):** needs design — the regex overload is distinguished by the FROM-pattern grammar, not arg types, so a name macro cannot disambiguate it from the FROM/FOR positional form. Fix is a functions.go transform: when substring is called with a single text pattern arg (SQL FROM-pattern form), rewrite to regexp_extract(string, pattern). Remove the identity substring mapping.
- **Classify token:** SUBSTRING(
- **Qualify map:** n/a for macro — specialFunctions AST rewrite (no pgcatalog map entry).
- **Regression test:** Assert substring('Thomas' FROM 'o(.)') returns 'mas'-style PG extraction (regexp_extract path) and the positional substring('Thomas' FROM 4 FOR 3)='mas' still works.

### encode  — missing
- **Signature:** encode(data bytea, format text) -> text
- **Evidence:** $ duckdb -c "SELECT encode('abc'::bytea,'base64')" -> Binder Error: No function matches 'encode(BLOB, STRING_LITERAL)'. Candidate: encode(VARCHAR)->BLOB (DuckDB's encode is single-arg, opposite direction). Native to_base64('abc'::bytea)->'YWJj'; hex('abc'::bytea)->'616263'. functions.go maps encode->encode identity (line 122), but the 2-arg PG form cannot resolve.
- **Impact:** Any client/ORM rendering bytea columns as text via encode(col,'base64'|'hex'|'escape') gets a hard Binder Error. DuckDB's single-arg encode points the opposite way, so the PG call never resolves. Affects binary-column serialization in standard PG SQL.
- **Fix (macro):** CREATE OR REPLACE MACRO encode(data, fmt) AS CASE lower(fmt) WHEN 'base64' THEN to_base64(data) WHEN 'hex' THEN lower(hex(data)) WHEN 'escape' THEN CAST(decode(data) AS VARCHAR) ELSE error('unsupported encode format: ' || fmt) END  -- 2-arg overload. NOTE DuckDB lacks macro overloading, so this 2-arg macro shadows the 1-arg builtin; remove the identity functions.go entry and ensure any single-arg encode use is migrated, or name the compat wrapper distinctly and rewrite in functions.go.
- **Classify token:** ENCODE(
- **Qualify map:** "encode": true,  // 2-arg PG bytea->text encoder (base64/hex/escape) macro
- **Regression test:** Assert encode('abc'::blob,'base64')='YWJj' and encode('abc'::blob,'hex')='616263'.

### width_bucket  — missing
- **Signature:** width_bucket(operand numeric, low numeric, high numeric, count int) -> integer
- **Evidence:** $ duckdb -c "SELECT width_bucket(5.35,0.024,10.06,5)" -> Catalog Error: Scalar Function with name width_bucket does not exist! Did you mean time_bucket?. Only time_bucket and equi_width_bins exist; no 4-arg numeric width_bucket. functions.go maps width_bucket->width_bucket identity (line 55) — a no-op that does not create the missing function. Expected result 3. DuckDB v1.5.2.
- **Impact:** BI/analytics tools (Metabase, Superset, Tableau) and dbt analytics macros emit the ANSI/PG-standard width_bucket(operand,low,high,count) for equi-width histogram binning; every such query hard-errors with a Catalog Error rather than returning the bucket index. The existing identity mapping is dead weight.
- **Fix (macro):** CREATE OR REPLACE MACRO width_bucket(operand, low, high, cnt) AS CASE WHEN operand < low THEN 0 WHEN operand >= high THEN cnt + 1 ELSE CAST(floor((operand - low) / ((high - low) / cnt)) AS INTEGER) + 1 END  -- matches PG equi-width semantics incl. below-range (0) and at/above-range (count+1) edges. Remove the identity functions.go entry so the macro is reached.
- **Classify token:** WIDTH_BUCKET(
- **Qualify map:** "width_bucket": true,  // SQL-standard equi-width histogram bucketing macro
- **Regression test:** Assert width_bucket(5.35,0.024,10.06,5)=3, width_bucket(-1,0,10,5)=0 (below), and width_bucket(10,0,10,5)=6 (at/above high).
